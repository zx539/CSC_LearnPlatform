## 2. 系统总览

### 2.1 业务目标

系统面向高校学习场景，提供从输入学习需求到持续跟踪反馈的闭环能力：

1. 对话式学习画像生成  
2. 多类型个性化学习资源生成  
3. 学习路径分阶段规划  
4. 进度/测试问卷驱动的学习评估  
5. 智能辅导问答  
6. 用户中心与账号安全管理

### 2.2 核心技术栈

| 层级 | 技术与组件 |
|---|---|
| 后端 | Python 3 + Flask |
| 智能服务 | `MultiAgentLearningSystem` + `SparkClient` |
| 前端 | HTML + CSS + 原生 JavaScript |
| 内容渲染 | marked + KaTeX + Mermaid + highlight.js |
| 数据持久化 | 文件系统 JSON（按用户隔离） |
| 模型支持 | Spark Ultra/Lite、豆包 Pro/Lite（统一封装） |

### 2.3 项目核心目录

```text

  services/
    multi_agent.py            # 多智能体编排（画像/资源/路径/问卷/评估/辅导）
    spark_client.py           # 模型客户端（重试、并发、QPS、流式处理）
    user_store.py             # 用户与学习记录存储、头像与密保能力

```

## 3. 总体架构

### 3.1 分层架构

1. **表现层（Frontend）**：采集输入、展示报告、提交问卷、轮询异步任务、渲染 Markdown/公式/导图。  
2. **应用层（Flask API）**：会话鉴权、参数校验、请求编排、错误映射、任务生命周期管理。  
3. **智能服务层（MultiAgent + SparkClient）**：多智能体职责拆分与模型调用治理。  
4. **数据层（UserStore + 文件系统）**：用户数据、运行结果、问卷历史、头像文件、异步任务状态持久化。

### 3.2 关键设计原则

1. **结构化优先**：模型输出优先约束为 JSON，降低后续解析复杂度。  
2. **双层纠偏**：后端生成纠偏 + 前端渲染前归一化，提升异常输入容错。  
3. **数据隔离**：用户数据目录独立于仓库，支持生产环境挂载与迁移。  
4. **可替换模型**：统一模型调用协议，降低供应商绑定风险。  
5. **慢任务异步化**：长链路生成采用后台任务，避免 HTTP 长连接超时。

## 4. 异步生成任务机制（重点优化）

### 4.1 背景与问题

学习方案生成链路包含多个模型调用阶段（画像、资源、路径、问卷），在豆包模型下可能持续较长时间。  
同步 `POST /api/generate` 容易受到网关 504 或连接超时影响，导致“模型已消耗 token 但前端失败”。

### 4.2 机制设计

系统将生成流程改为“提交任务 + 状态轮询”：

1. 前端发起 `POST /api/generate`，后端立即返回 `task_id`。  
2. 后端在后台线程执行完整生成流程。  
3. 前端轮询 `GET /api/generate/<task_id>` 获取状态。  
4. 状态为 `succeeded` 后返回完整报告；`failed` 返回错误信息。

### 4.3 状态模型

| 状态 | 含义 |
|---|---|
| `queued` | 任务已入队，等待执行 |
| `running` | 后台正在调用模型生成 |
| `succeeded` | 任务完成，可获取完整结果 |
| `failed` | 任务失败，返回错误原因 |

### 4.4 持久化与多进程一致性

为避免多 worker 部署时出现“轮询到不同 worker 找不到任务”，任务状态采取双写：

1. 进程内内存映射（快速读写）  
2. 磁盘 JSON 文件（`USER_DATA_DIR/_async_tasks`）持久化

读取逻辑先查内存，不命中则回源文件；完成态任务按 TTL 清理。  
关键配置：

- `GENERATE_TASK_WORKERS`：后台任务线程数（默认 2）  
- `GENERATE_TASK_TTL_SECONDS`：任务保留时间（默认 86400）

### 4.5 前端交互语义

1. 用户点击“生成”后立即看到“任务已提交”。  
2. 前端持续轮询并展示阶段状态文案。  
3. 用户点击取消时仅停止等待，不中断后台任务；完成后仍可在“已有学习项目”查看结果。

## 5. 后端模块设计

### 5.1 `backend/app.py`

职责：

1. 路由定义与登录态鉴权  
2. 生成任务创建、调度、状态查询  
3. 学习进度与测试问卷提交流程  
4. 用户中心、头像、密码与密保接口

关键内部函数：

- `normalize_model` / `resolve_model_config_path`：模型标识归一化与配置文件映射  
- `normalize_form_template`：问卷字段统一化  
- `extract_profile_core`：兼容历史画像结构  
- `_run_generate_task`：后台执行完整学习方案生成

### 5.2 `backend/services/multi_agent.py`

职责拆分：

1. **画像智能体**：从对话中提取学习画像  
2. **资源智能体**：生成课程讲解、导图、练习、案例、阅读、视频建议  
3. **路径智能体**：输出阶段目标、行动和检查点  
4. **问卷智能体**：生成进度问卷与测试问卷  
5. **评估智能体**：输出阶段评估、风险、下阶段建议  
6. **辅导智能体**：问答式支持

鲁棒性机制：

- JSON 失败时二次严格提示重试  
- 题型修复（`_repair_question_type`）  
- Markdown/Mermaid/数学公式归一化处理

### 5.3 `backend/services/spark_client.py`

主要能力：

1. 模型 URL 与鉴权加载（环境变量优先）  
2. QPS 与并发控制（信号量 + 节流）  
3. 超时、重试、退避  
4. 429/503/504 等过载场景识别  
5. 流式与非流式响应统一解析  
6. 非 JSON/HTML 异常响应识别与提示

### 5.4 `backend/services/user_store.py`

主要能力：

1. 用户登录注册与密码哈希  
2. 密保问题与答案哈希校验  
3. 学习运行记录持久化（run 数据）  
4. 头像保存、格式校验与读取  
5. 用户数据目录迁移与统计信息聚合

## 6. 前端模块设计

### 6.1 `frontend/static/js/app.js` 核心职责

1. 全局状态管理（模式、步骤、当前 run、问卷状态、辅导上下文）  
2. API 调用封装与异常处理  
3. 异步任务轮询（`waitGenerateTask`）  
4. Markdown/公式/导图渲染  
5. 用户中心与问卷交互流程

### 6.2 关键交互流程（生成）

1. 收集 `course/topic/dialogue/model`  
2. 调用 `POST /api/generate` 获取 `task_id`  
3. 轮询 `GET /api/generate/<task_id>`  
4. 成功后渲染报告并刷新项目列表

## 8. 数据模型与文件组织

### 8.1 用户目录

每个用户独立目录，典型内容：

1. 账号与基础信息  
2. 密保配置  
3. 学习运行记录（run）  
4. 头像文件

### 8.2 运行记录（Run）关键字段

| 字段 | 说明 |
|---|---|
| `run_name` | 项目标识 |
| `request` | 原始输入（课程、主题、模型等） |
| `report` | 画像、资源、路径、问卷、评估 |
| `report_markdown` | 各报告模块 Markdown |
| `progress_checkins` | 历史问卷与评估记录 |
| `output_dir` | 对应输出目录 |

### 8.3 异步任务文件

路径：`USER_DATA_DIR/_async_tasks/<task_id>.json`  
内容：状态、用户、时间戳、结果或错误信息。

## 9. 安全设计

### 9.1 账号与密保

1. 密码哈希存储，不落地明文  
2. 密保答案哈希校验  
3. 重置密码与改密路径分离

### 9.2 文件上传

头像上传包含多层校验：

1. Data URL 格式  
2. Base64 大小限制  
3. 解码后体积限制  
4. 文件签名（魔数）验证  
5. 路径安全与文件名约束

### 9.3 接口访问控制

核心业务接口统一通过会话校验；任务查询接口额外校验 `task_id` 所属用户，避免越权读取。

## 10. 稳定性与性能策略

### 10.1 稳定性

1. 模型调用超时、重试、退避  
2. QPS 与并发节流  
3. 题型与模板归一化  
4. 非 JSON/HTML 异常响应拦截  
5. 异步任务化降低长连接中断风险

### 10.2 性能

1. 资源生成与问卷生成按配置并发  
2. 前端轮询间隔自适应（避免高频打点）  
3. 后台任务线程数可通过环境变量调优

## 11. 配置与部署建议

### 11.1 关键环境变量

| 分类 | 变量示例 |
|---|---|
| 数据目录 | `USER_DATA_DIR` |
| 异步任务 | `GENERATE_TASK_WORKERS` `GENERATE_TASK_TTL_SECONDS` |
| 模型通用 | `SPARK_*` |
| 豆包模型 | `DOUBAO_*` `DOUBAO_PRO_*` `DOUBAO_LITE_*` |
| 输出并发 | `RESOURCE_MAX_PARALLEL` `FORM_MAX_PARALLEL` |

### 11.2 生产部署建议

1. 用户数据目录与输出目录置于仓库外独立挂载  
2. 使用 Gunicorn + Nginx 部署  
3. 即使已异步化，仍建议设置合理网关超时，避免其他慢接口受影响  
4. 对任务目录和用户目录配置最小权限

## 12. 已知边界与扩展方向

### 12.1 已知边界

1. 当前任务队列为进程内线程池 + 文件持久化，适合中小规模。  
2. 大规模并发场景建议迁移到专用任务队列（如 Celery/RQ）与数据库状态存储。

### 12.2 推荐扩展

1. 引入数据库统一存储 run 与任务状态  
2. 增加任务进度分阶段上报（非仅状态）  
3. 增加后台任务取消能力  
4. 建立自动化测试（接口、问卷一致性、异步任务生命周期）

## 13. 结语

当前版本已形成“多智能体学习闭环 + 工程化异步任务”的完整方案。  
其核心价值不仅是生成内容，更在于稳定地承载长链路学习编排，确保在真实模型时延下仍可用、可追踪、可迭代。
